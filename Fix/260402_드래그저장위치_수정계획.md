# 이미지 드래그 저장 위치 오류 수정계획

## 문제 현상
- 복사된 이미지를 바탕화면으로 드래그해도 프로젝트 폴더(탐색기 창)에 저장됨
- 드래그 후 `[Monitor] 새 항목: '[이미지]'` 메시지 발생 — 클립보드에 이미지가 다시 올라감
- 재현 조건: 항상 발생 (탐색기 창이 화면에 열려 있을 때)

## 원인 분석

### 1순위: `_save_image_to_folder` silent fail → 클립보드 fallthrough

- 파일: `pasteflow/main.py` `_on_drag_to_app()` L591-597
- `_save_image_to_folder` 호출이 `try/except: pass`로 감싸져 예외가 조용히 무시됨
- 저장 실패 시 `else: return` 분기를 건너뛰고 아래 `self.interceptor._set_clipboard(full_item)` 호출
- `_set_clipboard_ctypes`에서 `set_self_triggered(0.5)` 설정 후 이미지를 클립보드에 씀
- 이미지가 큰 경우 클립보드 쓰기가 0.5초를 초과 → `_ignore_until` 만료 후 Monitor가 재감지
- 결과: `[Monitor] 새 항목: '[이미지]'` 발생 + 이미지가 큐에 중복 추가

저장 실패의 실제 원인(현재 로그 없어 추정):
- **CF_DIB 고급 포맷(BITMAPV4/V5HEADER)**: `bih_size`가 108 또는 124인 경우 `_save_image_to_folder`의 CF_DIB 처리 코드가 PIL 파싱 실패 발생 가능
- **Google Drive 경로 쓰기 지연**: `G:\내 드라이브\...` 경로는 Google Drive 마운트 경로로 일시적 I/O 오류 발생 가능

### 2순위: `WindowFromPoint`가 Explorer 창 반환 → 사용자가 의도한 바탕화면 미감지

- 파일: `pasteflow/main.py` `_on_drag_to_app()` L567-577
- 바탕화면 `Progman`/`WorkerW` 는 z-order 최하위 → 그 위에 Explorer 창이 있으면 `WindowFromPoint`가 Explorer를 반환
- 로그: `root_class='CabinetWClass'` → 바탕화면 경로(`_DESKTOP_CLASSES`)가 아닌 Explorer 경로(`_EXPLORER_CLASSES`) 처리
- 사용자 화면에서 탐색기 창이 cursor 위치(564, 518)에 있어 바탕화면 드래그가 탐색기에 착지

### 3순위: 저장 성공/실패 피드백 없음

- 어디에 저장됐는지, 저장에 실패했는지 사용자가 알 수 없음
- 결과: 바탕화면에 없는 파일을 찾느라 혼란 발생

### Phase 1 — silent fail 제거 + 저장 진단 로그 추가

**목표**: `_save_image_to_folder` 실패 원인을 로그로 확인 가능하게 만들고,
fallthrough 시 `self_triggered` 타이머를 2.0초로 연장해 Monitor 재감지 방지.

**`_on_drag_to_app` 수정**:
```python
if folder:
    try:
        saved_path = _save_image_to_folder(full_item.image_data, folder)
        print(f"[DBG] 이미지 저장 성공: {saved_path!r}")
    except Exception as e:
        print(f"[DBG] 이미지 저장 실패: {e!r}")
        # fallthrough: 클립보드 경로로 진행, self_triggered 타이머 연장
        if self.interceptor.monitor:
            self.interceptor.monitor.set_self_triggered(2.0)
    else:
        return
```

**`_save_image_to_folder` 반환값**: 이미 `return path`가 있으므로 추가 작업 불필요.

**⚠️ `set_self_triggered(2.0)` 실행 순서 주의**:
`_set_clipboard_ctypes` 내부에서 `set_self_triggered(0.5)`를 다시 호출하므로,
`set_self_triggered(2.0)`은 반드시 `_set_clipboard` 호출 **이후**에 다시 설정해야 2.0초가 실제로 적용된다.

```python
if folder:
    try:
        saved_path = _save_image_to_folder(full_item.image_data, folder)
        print(f"[DBG] 이미지 저장 성공: {saved_path!r}")
    except Exception as e:
        print(f"[DBG] 이미지 저장 실패: {e!r}")
        # fallthrough 후 self_triggered 재설정 (아래 _set_clipboard에서 0.5초로 덮어쓰이므로 이후에 연장)
    else:
        return

# _set_clipboard 호출 (이미지 저장 실패 경로 + 텍스트/기타 항목)
self.interceptor._set_clipboard(full_item)
# 이미지 fallthrough 경우 클립보드 쓰기 완료 후 2.0초로 연장
if full_item.content_type == "image" and self.interceptor.monitor:
    self.interceptor.monitor.set_self_triggered(2.0)
```

- [x] 커밋: `Fix: 이미지 저장 실패 로그 추가 및 self_triggered 타이머 연장`

### Phase 2 — BITMAPV4/V5HEADER CF_DIB 지원

**현재 CF_DIB 처리 코드의 문제**:
```python
pixel_offset = 14 + bih_size + n_colors * 4
```
`bih_size`가 40(BITMAPINFOHEADER)일 때는 정상이지만,
108(BITMAPV4HEADER) 또는 124(BITMAPV5HEADER)일 때는 `n_colors` 계산과
`pixel_offset` 이 맞지 않아 PIL 파싱 실패 가능.

**수정**: `bih_size`에 따른 분기 없이 PIL에 전달하는 방식으로 단순화:
```python
else:
    # CF_DIB: BMP 파일 헤더(14바이트) 조립
    # pixel_offset은 bih_size 크기에 무관하게 파일 전체 크기 기준으로 계산
    bih_size = struct.unpack_from('<I', image_data, 0)[0]
    bit_count = struct.unpack_from('<H', image_data, 14)[0]
    if bit_count <= 8:
        colors_used = struct.unpack_from('<I', image_data, 32)[0]
        n_colors = colors_used if colors_used > 0 else (1 << bit_count)
    else:
        n_colors = 0
    pixel_offset = 14 + bih_size + n_colors * 4
    file_size = 14 + len(image_data)
    file_header = struct.pack('<2sIHHI', b'BM', file_size, 0, 0, pixel_offset)
    img = Image.open(io.BytesIO(file_header + image_data))
    img.save(path, 'PNG')
```

**⚠️ Phase 2 진행 조건**: Phase 1 로그에서 `[DBG] 이미지 저장 실패: ...` 메시지가 실제로 나타나고
PIL 파싱 관련 예외임이 확인된 경우에만 진행. 현재 코드의 CF_DIB 처리(`bit_count > 8` 시
`n_colors = colors_used = 0` → `pixel_offset = 14 + bih_size`)는 V4/V5 헤더와도 동일하게 적용되므로
Phase 1 로그 없이는 실질적 변경 사항이 없음.

### Phase 3 — 바탕화면 드래그 재현 조건 명확화 후 대응

**Phase 1 이후 재설계 필요**: Phase 3는 Phase 1 로그 결과에 따라 방향이 달라진다.

**시나리오 A — `_save_image_to_folder` 실패가 원인**:
- Phase 1, 2 수정으로 저장 성공 시 `return` → Monitor 재감지 사라짐
- 바탕화면에 드래그해도 탐색기 창이 있으면 탐색기 폴더에 저장되는 것은 **정상 동작**
  (cursor 위치에 탐색기 창이 실제로 존재하기 때문)
- 이 경우 Phase 3 불필요 — 사용자에게 "탐색기 닫고 드래그하면 바탕화면에 저장" 안내로 충분

**시나리오 B — 탐색기가 없는데 CabinetWClass 감지**:
- 탐색기 창 없이 바탕화면에 드래그했는데 여전히 CabinetWClass가 감지되는 경우
- 이때는 아래 `_get_explorer_folder` None 반환 fallback이 의미 있음

**⚠️ Progman fallback 금지 조건**: `_get_explorer_folder`가 유효한 folder 경로를 반환한 경우에는
Progman fallback 절대 불가 — 사용자가 탐색기 창 위에 의도적으로 드롭한 케이스를 바탕화면으로 잘못 처리하게 됨

**Phase 1 결과 확인 후 아래 중 하나 선택**:

```
시나리오 A → Phase 3 Skip
시나리오 B → 아래 대응 진행
```

**시나리오 B 수정 방향 (Phase 1 후 확정)**:
- `_get_explorer_folder(root_hwnd)` 반환값이 None이고 `root_class in _EXPLORER_CLASSES`인 경우만 Progman fallback 시도
- Progman 영역 확인은 `FindWindowEx("Progman", "SHELLDLL_DefView", ...)` 등 더 정밀한 방법 사용
