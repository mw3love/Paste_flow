## Critical

### C-1. `WA_ShowWithoutActivating` 제거 시 패널 포커스 탈취 위험

- **대상 Phase**: Phase 1
- **위치**: `image_preview.py` `__init__()` — `WA_ShowWithoutActivating` 제거

계획서는 "포커스를 받도록" 이 속성을 제거한다고 명시한다. 그러나 `ImagePreviewPopup`은 **패널이 활성화된 채로** 더블클릭 시 열린다. `WA_ShowWithoutActivating`을 제거하면 미리보기 팝업이 열릴 때 Qt가 자동으로 활성 창을 이 팝업으로 전환한다. 결과적으로:

1. 패널(`ClipboardPanel`)이 포커스를 잃고 `deactivate` → `hide()` 로직이 트리거될 수 있다 (`panel.py` L1307 `self.hide()`).
2. 더블클릭 → 팝업 오픈과 동시에 패널이 사라지는 현상 발생 가능.
3. Phase 3(다중 인스턴스)에서 팝업을 여러 개 열면 팝업들이 서로 포커스를 빼앗는 연쇄 반응 위험.

**권장 조치**: `WA_ShowWithoutActivating`을 유지하되, ESC 키 이벤트는 `keyPressEvent` 재정의 대신 `QShortcut(QKeySequence.StandardKey.Close, self)`를 통해 포커스 없이도 수신하는 방식을 검토. 또는 팝업 닫기 버튼(Phase 4)을 Phase 1로 앞당겨 ESC 없이도 닫을 수 있는 경로를 먼저 제공.

---

### C-2. `keyPressEvent`가 실제로 동작하지 않을 가능성 (ToolTip 계열 윈도우)

- **대상 Phase**: Phase 1
- **위치**: `image_preview.py` `keyPressEvent`

윈도우 플래그를 `Tool | FramelessWindowHint | WindowStaysOnTopHint`로 변경한다고 해도, `QLabel`을 직접 최상위 창으로 사용하는 구조에서 `keyPressEvent`가 호출되려면 창이 **키보드 포커스를 실제로 획득**해야 한다. C-1과 반대로, `WA_ShowWithoutActivating`을 제거해야 포커스를 받는다. 즉 C-1 문제와 ESC 동작 요건이 **서로 상충**한다.

- `WA_ShowWithoutActivating` 유지 → 패널 안전하지만 ESC 불가
- `WA_ShowWithoutActivating` 제거 → ESC 가능하지만 패널 포커스 탈취

계획서는 이 트레이드오프를 인식하지 못한 채 단순 제거를 지시하고 있어, Phase 1 완료 후 ESC가 동작하는 경우에도 패널 동작이 깨질 위험이 있다.

**권장 조치**: `installEventFilter`를 사용하여 전역 키 이벤트를 포커스 없이 수신하는 방식, 또는 팝업에 `activateWindow()`를 명시적으로 호출하고 패널의 `deactivate → hide` 조건을 이미지 팝업 활성 중에는 억제하는 방식 중 하나를 선택해야 한다. 계획서에서 이 분기 처리가 누락되어 있다.

## High

### H-1. `_instances` 리스트 메모리 누수 — `closeEvent` 미구현 시

- **대상 Phase**: Phase 3
- **위치**: `image_preview.py` — `_instances: list[ImagePreviewPopup]`

계획서는 `closeEvent`에서 `_instances`에서 자신을 제거한다고 명시하지만, `QLabel` 기반 최상위 창은 사용자가 OS 창 닫기 버튼이 아니라 `hide()`만 호출할 경우 `closeEvent`가 발생하지 않는다. Phase 1~2의 ESC(`self.close()`)는 `closeEvent`를 정상 트리거하지만, `hide_preview()`(현재 `panel.py` L253에서 드래그 중 호출)는 `hide()`를 사용하므로 `_instances`에서 제거되지 않는다.

결과: 드래그 중 숨겨진 팝업들이 `_instances`에 누적 → 앱 장시간 사용 시 메모리 증가.

**권장 조치**: `hide_preview()` 내에서 `close()` 호출로 일원화하거나, `hideEvent`에서도 `_instances` 제거를 수행하는 방어 코드를 추가해야 한다.

---

### H-2. `panel.py`의 싱글톤 참조 `ImagePreviewPopup.instance()` 잔존

- **대상 Phase**: Phase 3
- **위치**: `panel.py` L253 — `ImagePreviewPopup.instance().hide_preview()`

계획서 Phase 3에서 `instance()` / `toggle_preview()`의 "호환성 유지 또는 제거"를 검토 항목으로만 언급하고 체크박스 작업으로 명시하지 않았다. 그러나 `panel.py` L253은 드래그 중 항상 `ImagePreviewPopup.instance()`를 호출한다. 다중 인스턴스 전환 후 이 코드가 수정되지 않으면:

- 싱글톤 `_instance`가 제거된 경우 `AttributeError` 또는 `None` 참조 오류 발생
- 싱글톤을 남겨두면 `_instance`와 `_instances` 두 경로가 공존하여 상태 불일치

**권장 조치**: Phase 3 체크박스에 `panel.py` L253 수정(`hide_preview()` → `close_all()` 또는 동등 처리)을 명시적으로 추가해야 한다.

---

### H-3. `wheelEvent` 중 창 크기와 화면 경계 처리 누락

- **대상 Phase**: Phase 2
- **위치**: `image_preview.py` `_apply_scale()`

계획서의 `_apply_scale()`은 `setPixmap` + `adjustSize`만 수행한다. 그러나 줌 후 창이 확대되면 현재 창 위치에서 화면 밖으로 벗어날 수 있다. `show_preview()`의 화면 경계 조정 로직은 최초 표시 시에만 실행되므로, 줌 후에는 재실행되지 않는다.

- 8.0× 최대 배율에서 원본 640×480 이미지라면 창이 5120×3840으로 확대됨 (스케일 상한 없이)
- `adjustSize()`는 `QLabel`의 sizeHint 기반이므로 실제로는 원본 픽스맵 크기까지 팽창

**권장 조치**: `_apply_scale()` 내에 최대 출력 크기 상한(예: 화면 크기의 80%)을 적용하고, 창이 화면 밖으로 벗어나면 위치를 재조정하는 로직을 추가해야 한다. 계획서에 이 내용이 누락되어 있다.

## Medium

### M-1. Phase 순서 — 닫기 버튼(Phase 4)이 너무 늦음

- **대상 Phase**: Phase 4 전체

`WA_ShowWithoutActivating`을 제거(Phase 1)하면 팝업이 포커스를 받게 되어 ESC로 닫을 수 있다. 그러나 C-1에서 설명한 패널 포커스 탈취 문제 때문에 실제로는 `WA_ShowWithoutActivating`을 유지해야 할 가능성이 높고, 그 경우 Phase 1 완료 후 팝업을 닫을 방법이 없다.

닫기 버튼은 포커스와 무관하게 마우스 클릭만으로 동작하므로, **Phase 4를 Phase 1로 병합**하거나 Phase 1보다 먼저 구현하는 것이 안전하다. 현재 순서는 Phase 1~3 기간 동안 "닫기 수단 없는 팝업"이 존재하는 기간을 만든다.

---

### M-2. `QLabel` 단일 상속의 한계 — 복잡도 증가 시 리팩토링 필요 가능성

- **대상 Phase**: Phase 1~4 전체

현재 `ImagePreviewPopup`은 `QLabel`을 직접 상속한다. 닫기 버튼 오버레이(Phase 4), 줌 레이블, 드래그 핸들 등이 추가되면 `QLabel` 단일 위젯으로는 레이아웃 구성이 어색해진다. `QWidget` + 내부 `QLabel`(이미지) 구조가 더 자연스럽다.

계획서가 현재 구조를 그대로 유지한다고 가정하므로, Phase 4의 닫기 버튼을 `QLabel` 오버레이로 구현하는 방법(계획서 언급)은 절대 좌표 기반 수동 배치가 필요해 `resizeEvent`에서 위치를 재계산해야 하는 추가 코드가 발생한다.

**권장 조치**: 위험은 낮지만, Phase 3 전후로 `QWidget` 기반으로 전환하는 리팩토링을 검토하는 것이 Phase 4 구현을 단순하게 만든다.

---

### M-3. 다중 미리보기 창의 초기 위치 겹침

- **대상 Phase**: Phase 3
- **위치**: `open_new()` 클래스메서드

계획서의 `open_new()`는 `global_pos`를 그대로 전달한다. 같은 항목을 컨텍스트 메뉴 → 미리보기로 두 번 열거나, 비슷한 위치에서 여러 이미지를 빠르게 열면 모든 창이 동일한 좌표에 생성되어 완전히 겹친다. 사용자는 창이 하나만 열렸다고 착각할 수 있다.

**권장 조치**: `open_new()` 내에서 기존 `_instances`의 개수에 따라 위치를 오프셋(예: 20px씩 cascade)하는 로직을 추가해야 한다. 계획서에 이 케이스가 누락되어 있다.

## Low

### L-1. `_to_png()` 변환 예외가 조용히 무시됨 (기존 코드 문제, 계획서 미언급)

- **대상 Phase**: 해당 없음 (기존 코드)
- **위치**: `image_preview.py` L89~97

`_to_png()`는 예외 발생 시 `None`을 반환하고, `show_preview()`는 `None`이면 조용히 리턴한다. 휠 줌(Phase 2)에서 원본 픽스맵을 저장하는 `_original_pixmap`은 변환 실패 시 설정되지 않아 이후 `_apply_scale()`에서 `AttributeError` 가능성이 있다. 계획서가 이 방어 코드를 명시하지 않는다.

**권장 조치**: `_apply_scale()` 내에 `hasattr(self, '_original_pixmap')` 가드 추가를 Phase 2 체크박스에 포함시켜야 한다.

---

### L-2. `QApplication.primaryScreen()` — 다중 모니터 환경에서 부정확

- **대상 Phase**: Phase 1 (최대 크기 변경) / Phase 2 (줌)
- **위치**: `image_preview.py` `show_preview()` L64

`primaryScreen()`은 항상 주 모니터 기준이다. 팝업이 보조 모니터에 열릴 경우 화면 경계 계산이 틀려 창이 잘릴 수 있다. `QApplication.screenAt(global_pos)`를 사용하면 팝업이 열리는 모니터 기준으로 경계를 계산할 수 있다. 계획서의 640×480 상한 확대(Phase 1) 시 이 문제가 더 빈번해진다.

이 문제는 기존 코드에도 존재하지만, 계획서 범위 내에서 언급 및 수정이 권장된다.

## Phase 간 의존성 요약

```
Phase 1 (윈도우 타입 변경 + 드래그 + ESC)
  ↓ 전제조건: C-1/C-2 해결 방향 결정 필요 (WA_ShowWithoutActivating 유지/제거)
Phase 2 (휠 줌)
  ↓ Phase 1 완료 필수 (원본 pixmap 저장 구조 필요)
Phase 3 (다중 인스턴스)
  ↓ Phase 2 완료 필수 (_original_pixmap이 인스턴스 변수이므로 다중화 전 구조 확정)
  ↓ panel.py L253 동시 수정 필수 (H-2)
Phase 4 (닫기 버튼 + UI)
  ↓ Phase 3 완료 후 진행 — 단, M-1 사유로 닫기 버튼을 Phase 1로 앞당기는 것 권장
```

## 종합 의견

계획서의 접근법(윈도우 타입 변경 → 줌 → 다중 창 순서)은 논리적으로 올바르다. 그러나 `WA_ShowWithoutActivating` 제거로 인한 포커스 탈취 문제(C-1, C-2)가 Phase 1 완료 직후 패널 동작을 파괴할 수 있어, 이 부분에 대한 처리 방향을 **Phase 1 시작 전에 명확히 결정**해야 한다. 나머지 High/Medium 항목들은 구현 중에 방어 코드를 추가하는 수준으로 대부분 해결 가능하다.

Critical 2건이 모두 Phase 1에 집중되어 있으므로, Phase 1 구현 전 이 리뷰 결과를 검토한 후 진행하는 것을 권장한다.
