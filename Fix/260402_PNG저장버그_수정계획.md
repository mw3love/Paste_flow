# 이미지 드래그 → PNG 저장 안됨 수정계획

## 문제 현상
- 패널에서 이미지 항목을 탐색기 폴더 또는 바탕화면으로 드래그해도 PNG 파일이 저장되지 않음
- 재현 조건: 항상 발생 (탐색기 드롭 모든 케이스)
- 사용자 목격 증상: 드래그 후 탐색기에 파일 없음, 조용한 실패

## 원인 분석

### 1순위: `_get_explorer_folder`에서 `CoInitialize` 호출이 예외를 발생시켜 항상 None 반환

- 파일: `pasteflow/main.py` `_get_explorer_folder()` L112-128
- Qt 메인 스레드는 이미 COM을 **MTA 모델**(`CoInitializeEx(COINIT_MULTITHREADED)`)로 초기화한 상태
- 이 상태에서 `pythoncom.CoInitialize()`(=`COINIT_APARTMENTTHREADED`)를 호출하면
  `RPC_E_CHANGED_MODE (0x80010106)` 예외가 발생
- 해당 예외가 외부 `except Exception: pass`에 잡혀 **항상 None 반환**
- `_on_drag_to_app`에서 `folder is None` → `if folder:` 분기 건너뜀 → 기존 클립보드 경로로 fall-through
- `_get_desktop_path`도 동일 패턴으로 동일하게 실패

```python
# 현재 구현 (버그)
def _get_explorer_folder(hwnd: int):
    try:
        pythoncom.CoInitialize()   # ← Qt MTA 스레드에서 STA 요청 → RPC_E_CHANGED_MODE 예외
        try:
            shell = win32com.client.Dispatch("Shell.Application")
            ...
        finally:
            pythoncom.CoUninitialize()
    except Exception:              # ← 예외를 조용히 삼킴
        pass
    return None                    # ← 항상 None

# 계획서 원본 설계 (정상)
def _get_explorer_folder(hwnd: int):
    try:
        import win32com.client      # CoInitialize 없음 — Qt 스레드 COM 그대로 사용
        shell = win32com.client.Dispatch("Shell.Application")
        ...
```

**비고**: 기존 계획서(`260402_이미지드래그저장_수정계획.md`)의 설계 코드에는 `CoInitialize` 없었으나
실제 커밋 시 추가된 것으로 추정.

### 2순위: `_on_drag_to_app`에서 `_save_image_to_folder` 예외 처리 없음

- 파일: `pasteflow/main.py` `_on_drag_to_app()` L454-456
- `_save_image_to_folder` 호출 주위에 try/except 없음
- PIL이 이미지 파싱에 실패하거나 권한 오류 시 조용히 실패
- 현재는 1순위 버그가 먼저 막으므로 이 경로가 실행되지 않아 드러나지 않음

### 3순위: CF_HDROP 이미지(JPEG 등) 저장 실패

- 파일: `pasteflow/main.py` `_save_image_to_folder()` L147-176
- 탐색기에서 JPEG 파일을 Ctrl+C 하면 `image_data`에 JPEG raw bytes 저장됨
- `_save_image_to_folder`는 `\x89PNG`가 아닌 경우 무조건 CF_DIB로 처리
- JPEG bytes + 가짜 BMP 헤더를 PIL에 전달 → PIL 파싱 실패 → 예외 발생
- 원인 2순위 버그와 결합하여 조용한 실패

## 단계별 수정 계획

### Phase 1 — CoInitialize 제거 (핵심 버그 수정)

**변경 내용**: `_get_explorer_folder`와 `_get_desktop_path`에서 `pythoncom.CoInitialize()`/`CoUninitialize()` 제거.
Qt 메인 스레드에서 COM은 이미 초기화되어 있으므로 `win32com.client.Dispatch`를 직접 호출 가능.

```python
# _get_explorer_folder 수정 후
def _get_explorer_folder(hwnd: int):
    """CabinetWClass HWND → 현재 폴더 경로 반환. 실패 시 None."""
    try:
        import win32com.client
        shell = win32com.client.Dispatch("Shell.Application")
        for window in shell.Windows():
            try:
                if int(window.HWND) == hwnd:
                    return window.Document.Folder.Self.Path
            except Exception:
                continue
    except Exception:
        pass
    return None

# _get_desktop_path 수정 후
def _get_desktop_path() -> str:
    """사용자 바탕화면 경로 반환."""
    try:
        import win32com.client
        return win32com.client.Dispatch("WScript.Shell").SpecialFolders("Desktop")
    except Exception:
        return os.path.expanduser("~/Desktop")
```

- [x] `_get_explorer_folder`: `import pythoncom` 제거, `CoInitialize()`/`CoUninitialize()` 제거, inner try/finally 제거
- [x] `_get_desktop_path`: `import pythoncom` 제거, `CoInitialize()`/`CoUninitialize()` 제거, inner try/finally 제거
- [x] 커밋: `Fix: _get_explorer_folder CoInitialize 제거 — Qt MTA 스레드 COM 충돌 해결`

**`_save_image_to_folder` 포맷 감지 강화**:

현재는 `\x89PNG`만 직접 처리하고 그 외는 CF_DIB로 간주함.
JPEG(`\xFF\xD8\xFF`), GIF(`GIF8`), WebP(`RIFF....WEBP`) 등은 PIL이 직접 열 수 있음.

```python
# 추가 포맷 magic bytes

    b'GIF8',            # GIF
    b'RIFF',            # WebP (RIFF....WEBP)
    b'\x89PNG',         # PNG
    b'BM',              # BMP 파일 (파일 헤더 있는 경우)
    b'\x00\x00\x01\x00',# ICO
)

def _save_image_to_folder(image_data: bytes, folder: str) -> str:
    ...
    # PNG 및 PIL 직접 인식 가능한 포맷
    if any(image_data.startswith(sig) for sig in _DIRECT_OPEN_SIGNATURES):
        img = Image.open(io.BytesIO(image_data))
        img.save(path, 'PNG')
    else:
        # CF_DIB raw → BMP 헤더 조립
        ...
```

### Phase 3 — 클립보드 캡처 실패 (EnumClipboardFormats 1418 에러)

**발견 경위**: `b9lehdevu.output` 로그 분석 (2026-04-02)

**문제 현상**:
- `_read_clipboard()` 내부에서 `OpenClipboard()` 성공 후 `EnumClipboardFormats()` 호출 시 error 1418 발생
- 이후 `finally`의 `CloseClipboard()`도 1418로 실패
- 결과: 예외가 `_on_clipboard_changed()`까지 전파 → `ClipboardItem` 미반환 → **이미지가 DB에 저장 안 됨**

**원인**:  
`GetClipboardData(CF_HDROP)` 호출 시 Explorer의 delayed rendering 또는 OLE 내부 로직이 클립보드 소유권을 중간에 가져가는 race 조건.  
또는 drag-out 중 PasteFlow가 클립보드에 쓸 때 WM_CLIPBOARDUPDATE 재진입.

**현재 코드 (버그)**:
```python
# clipboard_monitor.py L115~237
try:
    win32clipboard.OpenClipboard()   # 성공
except Exception:
    return None

try:
    # ... 포맷 읽기 ...
    fmt = 0
    while True:
        fmt = win32clipboard.EnumClipboardFormats(fmt)  # ← 1418 — 예외 전파
        ...
finally:
    win32clipboard.CloseClipboard()  # ← 1418 — 또 예외 (clipboard never closed)
```

**수정 방향**:
1. `OpenClipboard()` 성공 여부를 플래그(`_opened`)로 추적 → `CloseClipboard()`를 guard로 보호
2. `EnumClipboardFormats` 루프를 try-except로 감싸 loop break 처리 (캡처는 부분 성공으로 계속)

```python
# 수정 후
_opened = False
try:
    win32clipboard.OpenClipboard()
    _opened = True
except Exception:
    return None

try:
    # ... 포맷 읽기 ...
    fmt = 0
    while True:
        try:
            fmt = win32clipboard.EnumClipboardFormats(fmt)
        except Exception:
            break  # 클립보드 소유권 상실 시 loop 중단, 이미 읽은 데이터로 계속
        if fmt == 0:
            break
        ...
finally:
    if _opened:
        try:
            win32clipboard.CloseClipboard()
        except Exception:
            pass
```

- [x] `_read_clipboard()`: `_opened` 플래그 추가, `CloseClipboard()` guard 처리
- [x] `_read_clipboard()`: `EnumClipboardFormats` 루프 try-except 추가
- [x] 커밋: `Fix: _read_clipboard EnumClipboardFormats 1418 — CloseClipboard guard 및 루프 예외 처리`
- [ ] eval-plan evaluate 자동 실행 (HEAD~1)
